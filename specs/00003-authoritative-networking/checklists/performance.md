- [X] CHK016 Is the delta-encoding baseline defined precisely (delta vs each client's last-acknowledged snapshot, identified by `baseline_id`) so encoding correctness/efficiency is assertable? [Measurability, Spec §TR-013] <!-- Evaluator: Covered by spec.md §TR-013 + contracts/protocol.md §Snapshot (baseline_id) / §Reconciliation anchor + tightened §TR-045(a) -->
- [X] CHK017 Is the "unchanged entities cost ~1 bit" efficiency expectation stated as a checkable property of the delta encoding rather than an aspiration? [Measurability, contracts/protocol.md §EntityRecord] <!-- Evaluator: Resolved — added spec.md §TR-045(b): unchanged-entity ≤1 bit is a checkable encoding property asserted by a unit test, not an aspiration -->
- [X] CHK018 Are the quantized field ranges and bit-widths specified (position/velocity to sector-relative bounds, heading to fixed-bit angle) precisely enough that encoded size is deterministic? [Clarity, contracts/protocol.md §EntityRecord] <!-- Evaluator: Resolved — added spec.md §TR-045(c): field widths/ranges fixed in protocol::quantize so encoded size is deterministic per build; concrete numbers impl-pinned, precision tuning deferred to E009 -->
- [X] CHK019 Is an MTU bound on snapshot size stated as a requirement (snapshots not fragmented past MTU) so oversized snapshots are a defined failure rather than silent? [Completeness, research §Bandwidth baseline & snapshot model] <!-- Evaluator: Covered by spec.md §TR-029 (snapshot ≤ MTU, baseline ≤1200 B payload, no IP fragmentation) -->
- [X] CHK020 Does the spec define behavior when a delta snapshot would exceed the MTU (split, prioritize, drop) for the baseline session, or explicitly defer that to E009? [Edge-Case, contracts/protocol.md §Out of scope] <!-- Evaluator: Covered by spec.md §TR-029 (split across snapshots or drop lowest-priority entities; AOI prioritization deferred to E009) -->
- [X] CHK021 Is the quantization requirement traceable to a single authority (TR-013/AD-003/bitcode) so the baseline encoding is distinguished from E009's deferred quantization-budget tuning? [Traceability, Spec §Compliance Check] <!-- Evaluator: Covered by spec.md §TR-013 + plan §AD-003 + §Compliance Check advisory (baseline encoding vs E009 quantization-budget) + reinforced §TR-045(c) -->
- [X] CHK022 Is the snapshot ack/baseline mechanism (`SnapshotAck` → `baseline_id`) specified well enough to assert that a lost ack degrades gracefully (re-baseline) without unbounded snapshot growth? [Edge-Case, contracts/protocol.md §Reconciliation anchor] <!-- Evaluator: Resolved — added spec.md §TR-045(d): lost ack → delta vs last acked baseline (or full keyframe if none); growth bounded by MTU guard (TR-029) -->
